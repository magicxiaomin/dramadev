## Final recommendation

Approve the Q15 decision packet as architecture-safe proposal material only: keep `unlocked=1` as a fake-only review/demo signal for the current mock unlock/pass flow, with a bright-line rule that it must never grant real access, never become production entitlement, and must be ignored, stripped, disabled, or no-op outside explicitly approved fake/sandbox/review contexts once production entitlement exists.

This recommendation is safe because it preserves the PRD-required fake P0 same-episode return evidence without introducing backend, database, auth, login, payment, subscription, wallet ledger, Facebook API, analytics, production video, server entitlement, deployment, DNS, secrets, legal/compliance, brand, or NovelHub production infrastructure.

## Architecture summary

`unlocked=1` is acceptable only as an observable fake review signal in the P0 prototype boundary. It can tell reviewers that a mock single-episode unlock or mock Story Pass return landed on the same watch episode, but it cannot be an authorization source because public URL query params are user-editable.

The safe boundary is therefore: query-param state may affect fake/local review display in `/variant-b/watch/[showId]`; production access control must come from a production-controlled entitlement source outside the public URL. Future production entitlement may ignore, strip, disable, or no-op `unlocked=1`; this gate does not choose the mechanism.

The current repo stack already matches the expected P0 prototype shape: Next.js App Router, TypeScript, React, Tailwind, static fake data, and client-side mock state. Current code inspection found `parseWatchQueryParams` parses `unlocked` as a boolean, `buildWatchEpisodeHref` emits `unlocked=1` only when asked to build a mock unlocked return, `buildPassReturnHref` returns to the same watch episode with `unlocked=1`, and `getEpisodeLockState` treats `unlocked` as local display state after the `freeEpisodes` lock point. No backend entitlement, payment, subscription, login, Facebook API, production analytics, or real video infra is implied by this gate.

## P0 invariant preservation

Q15 preserves P0 if and only if these invariants remain true:

- Facebook ad traffic still lands directly on `/variant-b/watch/[showId]?episode=1&source=facebook`, not Home, Search, Browse/Genre, Show Detail, Pass, login, recharge, install education, entitlement, or payment.
- Free preview episodes still play before login, recharge, Story Pass promotion, PWA install prompt, payment-like prompt, real attribution processing, backend lookup, or production entitlement check.
- `story.freeEpisodes` remains the per-drama lock point source.
- First locked episode remains `episode > story.freeEpisodes && unlocked !== true` in the fake P0 boundary.
- The first locked episode still shows locked state and opens the Unlock Drawer; closing the drawer keeps the user on the locked episode.
- Single-episode unlock remains the first-lock primary CTA; Story Pass remains secondary.
- Fake single-episode unlock returns to `/variant-b/watch/[showId]?episode=[lockedEpisode]&unlocked=1` for the same show and episode.
- Fake Story Pass returns to `/variant-b/watch/[showId]?episode=[lockedEpisode]&unlocked=1` for the same show and episode.
- `unlocked=1` does not change showId, episode number, source/attribution context, starting episode, free chain order, drawer timing, CTA hierarchy, or Pass return semantics.
- `unlocked=1` remains fake review state only and cannot become production entitlement, authentication, payment proof, subscription state, wallet state, receipt state, or authorization.

## Guardrails and acceptance criteria

Required guardrails:

2. Treat `unlocked=1` as local prototype display state, not as a source of truth for access.
3. Do not persist `unlocked=1` as wallet, unlock history, subscription, login, receipt, user, or entitlement state.
