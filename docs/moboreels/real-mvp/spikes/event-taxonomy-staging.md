# Phase 4C Spike 5 — Event Taxonomy and Staging Analytics Contract

Status: draft spec for local/staging only
Source plan: `docs/moboreels/real-mvp/spike-plan.md` §4 Spike 5
PRD source of truth: `docs/moboreels/scene-flow-facebook-ad-conversion-prd.md`
Boundary: no Meta Pixel, no CAPI, no Meta/Facebook API, no production analytics endpoint, no backend/database, no real user/payment identifiers

## 1. Purpose

Define a safe local/staging event taxonomy for the SceneFlow Facebook-ad conversion funnel so later reviewers can verify event order, required properties, dedupe behavior, and consent/privacy gates without connecting any production analytics infrastructure.

This spec is a contract for fake/local capture only. It must not be treated as approval to add production analytics, Meta Pixel, CAPI, a data warehouse, a backend event collector, or a real identity graph.

## 2. Funnel invariant covered

The taxonomy must preserve the accepted P0 route and conversion invariant:

```txt
/variant-b/watch/[showId]?episode=1&source=facebook
-> free episode chain without login
-> first locked episode
-> Unlock Drawer
-> login/payment only after paid action, if simulated
-> fake grant/return simulation
-> same episode with unlocked=1 as UX state only
```

Events must never create a login wall, payment wall, PWA prompt, redirect to Home, or alternate conversion journey before free preview playback.

## 3. Allowed and disallowed capture targets

Allowed for Phase 4C:

- Browser console logging in local development.
- In-memory event buffer for the current page/session.
- Local/staging test fixture output such as a JSON file generated by tests.
- Staging-only debug panel or route if clearly labeled fake/staging.
- Test assertions that inspect emitted events without network egress.

Disallowed for Phase 4C:

- Meta Pixel, Facebook SDK, Conversions API, Marketing API, Graph API, or any Meta/Facebook production endpoint.
- Google Analytics, Segment, Amplitude, Mixpanel, PostHog, Plausible, Vercel Analytics, or any production third-party analytics endpoint.
- Backend/database event ingestion.
- Persistent production identifiers, real account IDs, payment IDs, email, phone, IP address, device fingerprint, or real ad-platform click IDs.
- Production dashboards, attribution optimization, retargeting audiences, or ad campaign feedback loops.

## 4. Naming conventions

- Event names use lowercase snake_case.
- Names describe user-visible funnel milestones, not implementation details.
- Event names are stable once reviewed; add new names rather than silently changing meaning.
- Properties use lowercase snake_case.
- Boolean properties use positive phrasing, e.g. `is_first_locked_episode`.
- Timestamp properties use ISO-8601 strings in UTC for local/staging fixtures; browser runtime may also include a monotonic debug timestamp for ordering.

## 5. Common event envelope

Every event should share this local/staging envelope:

| Property | Type | Required | Notes |
| --- | --- | --- | --- |
| `event_name` | string | yes | One of the names in §6. |
| `event_id` | string | yes | Local/staging UUID or deterministic test ID. Not a Meta event ID. |
| `event_version` | string | yes | Start with `phase4c.1`. |
| `occurred_at` | string | yes | UTC ISO timestamp or deterministic test timestamp. |
| `session_debug_id` | string | yes | Local/staging random ID. Must not be a real user/account/device identifier. |
| `route` | string | yes | Path only, e.g. `/variant-b/watch/midnight-lantern-oath`. Avoid full URL if it may contain unsafe params. |
| `show_id` | string | yes where applicable | Current show/story ID. |
| `episode` | number | yes where applicable | Current episode number. |
| `source` | string | optional | Expected P0 value: `facebook`; treat as self-reported query string, not verified attribution. |
| `attribution_debug` | object | optional | Safe allowlisted params only; see §7. |
| `surface` | string | yes | e.g. `watch`, `unlock_drawer`, `pass`, `fake_auth`, `fake_payment`. |
| `environment` | string | yes | Must be `local` or `staging`; never `production` in Phase 4C. |
| `is_mock` | boolean | yes | Must be `true` for Phase 4C. |
| `consent_state` | string | yes | One of `not_requested`, `declined`, `granted_staging`, `unknown`. No production consent posture is approved. |
| `dedupe_key` | string | yes | See §8. |

## 6. Event table

| Event | Trigger | Required properties | Optional safe properties | Notes |
| --- | --- | --- | --- | --- |
| `ad_landing_viewed` | Watch route loads from the P0 ad landing route. | common envelope, `show_id`, `episode`, `source`, `landing_episode` | `attribution_debug`, `entry_referrer_kind` | Must not fire on Home/Search first. For P0 route, `episode` should be `1`. |
| `free_episode_started` | A free episode begins playback/preview. | common envelope, `show_id`, `episode`, `free_episode_count`, `playback_state` | `autoplay` | Must be allowed without login/payment prompts. |
| `free_episode_completed` | User reaches the completion state for a free episode. | common envelope, `show_id`, `episode`, `next_episode`, `next_episode_is_locked` | `completion_method` | Supports free chain continuation measurement. |
| `locked_episode_viewed` | User reaches an episode where `episode > freeEpisodes` and no authoritative entitlement is present. | common envelope, `show_id`, `episode`, `free_episode_count`, `is_first_locked_episode` | `trigger_source` | Must not look like playback/network error. |
| `unlock_drawer_opened` | Unlock Drawer is shown for a locked episode. | common envelope, `show_id`, `episode`, `drawer_trigger`, `primary_cta`, `secondary_cta` | `mock_balance`, `mock_episode_cost` | Primary CTA must remain single-episode unlock; Story Pass secondary. |
| `unlock_cta_clicked` | User taps Unlock EP or Story Pass CTA. | common envelope, `show_id`, `episode`, `unlock_intent`, `cta_label` | `mock_balance`, `mock_episode_cost`, `mock_pass_cost` | This is the first acceptable point to simulate auth/payment interruption. |
| `auth_started` | Fake/staging auth interruption begins after paid action. | common envelope, `show_id`, `episode`, `unlock_intent`, `return_episode`, `return_drawer_state` | `fake_auth_provider` | No real OAuth/OTP/email provider. |
| `auth_returned` | Fake/staging auth returns success/cancel/failure. | common envelope, `show_id`, `episode`, `unlock_intent`, `auth_result`, `return_episode_matches` | `return_reason` | Failure/cancel must return to the same locked episode, not Home. |
| `fake_payment_started` | Fake payment simulator begins after unlock intent and optional fake auth. | common envelope, `show_id`, `episode`, `unlock_intent`, `fake_transaction_id`, `idempotency_key` | `mock_amount_coins` | Fake transaction ID must be local/staging only. |
| `fake_payment_succeeded` | Fake payment simulator reports success. | common envelope, `show_id`, `episode`, `unlock_intent`, `fake_transaction_id`, `idempotency_key` | `duplicate_of_event_id` | Must be safe for duplicate callback testing. |
| `fake_payment_failed` | Fake payment simulator reports cancel/failure. | common envelope, `show_id`, `episode`, `unlock_intent`, `fake_transaction_id`, `payment_result` | `failure_reason_code` | No real payment data or provider codes. |
| `entitlement_granted` | Fake entitlement grant is produced from successful fake payment/unlock. | common envelope, `show_id`, `episode`, `unlock_intent`, `fake_entitlement_id`, `grant_scope`, `idempotency_key` | `duplicate_grant` | URL `unlocked=1` is UX state only; it is not authoritative entitlement. |
| `same_episode_resumed` | User returns to or resumes the same locked episode after fake grant. | common envelope, `show_id`, `episode`, `unlock_intent`, `return_episode_matches`, `unlocked_query_state` | `resume_source` | Must verify user is not sent to Home and episode context is not lost. |

## 7. Safe attribution parameters

Only these query parameters may be copied into `attribution_debug` during Phase 4C:

- `source`
- `campaign_id`
- `adset_id`
- `ad_id`
- `creative_id`
- `placement`
- `utm_source`
- `utm_campaign`
- `utm_content`
- `utm_medium`
- `utm_term`

Rules:

- Treat all attribution params as untrusted strings.
- Store only allowlisted keys.
- Trim long values in local/staging fixtures, with a suggested maximum of 128 characters per value.
- Drop nested objects, arrays, scripts, URLs with credentials, and unknown query params.
- Do not collect Meta click IDs or identifiers unless a future legal/product review explicitly approves them.
- Do not use attribution params for production optimization, retargeting, or audience building in Phase 4C.

## 8. Dedupe proposal

### 8.1 Dedupe key format

For local/staging, each event should include:

```txt
{environment}:{event_name}:{session_debug_id}:{show_id}:{episode}:{dedupe_scope}:{dedupe_sequence_or_id}
```

Recommended `dedupe_scope` values:

- `route_view` for `ad_landing_viewed` and `locked_episode_viewed`.
- `playback_episode` for episode start/complete events.
- `drawer_open` for drawer open events.
- `cta_click` for CTA click events.
- `auth_attempt` for fake auth events.
- `fake_payment_attempt` for fake payment events.
- `fake_grant` for fake entitlement grant events.
- `resume` for same-episode resume events.

### 8.2 Event-specific dedupe notes

| Event | Dedupe recommendation |
| --- | --- |
| `ad_landing_viewed` | One per route mount per `session_debug_id` + `show_id` + `episode` + safe attribution hash. |
| `free_episode_started` | One per episode playback start attempt; replay after manual restart may increment sequence. |
| `free_episode_completed` | One per episode completion state transition. Guard against repeated completion overlay renders. |
| `locked_episode_viewed` | One per locked route entry. Reopen drawer separately with `unlock_drawer_opened`. |
| `unlock_drawer_opened` | One per distinct open action; include `drawer_trigger` so auto-open and tap-to-reopen can both be represented. |
| `unlock_cta_clicked` | One per user click/tap; do not dedupe away intentional repeated attempts after failure. |
| `auth_started` / `auth_returned` | Pair by local `auth_attempt_id` or shared `idempotency_key`. |
| `fake_payment_started` / result | Pair by `fake_transaction_id` and `idempotency_key`. Duplicate success should reference the original event or idempotency key. |
| `entitlement_granted` | Idempotent on `grant_scope` + `show_id` + `episode` or pass scope + `idempotency_key`. |
| `same_episode_resumed` | One per successful return/resume after fake grant. |

### 8.3 Idempotency and duplicate callback test cases

The staging contract must support these duplicate scenarios without double-counting grants:

1. Fake payment success emitted twice with the same `idempotency_key`.
2. Fake entitlement grant retried after a delayed callback.
3. User refreshes the unlocked return route with `unlocked=1`.
4. User closes and reopens the Unlock Drawer without starting a new purchase.

Expected behavior: retain separate debug events where useful for QA, but mark duplicate grant/payment outcomes with `duplicate_of_event_id` or `duplicate_grant: true`; do not create a second fake entitlement grant for the same idempotency key.

## 9. Consent and privacy notes

Phase 4C consent posture is a placeholder, not a legal conclusion.

Requirements:

- Default `consent_state` should be `not_requested` unless a staging-only toggle exists.
- No production tracking may occur when consent is unknown or absent.
- Do not collect direct identifiers: name, email, phone, real account ID, IP address, payment ID, exact device fingerprint, or real Facebook/Meta identifiers.
- Do not infer sensitive attributes from drama content, ad params, location, or behavior.
- Staging-only consent toggles must be labeled as non-production and must not become real cookie/banner logic without legal/product approval.
- Test fixtures should use synthetic values such as `session_debug_id: local-session-001`.

Future production analytics requires a separate approval gate for consent UX, retention, privacy policy copy, vendor contracts, and jurisdiction-specific rules.

## 10. Local/staging verification checklist

A reviewer should be able to verify the contract with local/staging evidence before any implementation plan is approved:

- [ ] Load `/variant-b/watch/[showId]?episode=1&source=facebook` and observe `ad_landing_viewed` without any production analytics network call.
- [ ] Start EP1 and observe `free_episode_started` without login or payment prompts.
- [ ] Complete each free episode and observe `free_episode_completed` with `next_episode_is_locked` set correctly.
- [ ] Enter the first locked episode and observe `locked_episode_viewed` plus `unlock_drawer_opened`.
- [ ] Close/reopen the drawer and verify duplicate drawer events are scoped to open actions, not fake grants.
- [ ] Click single-episode unlock and observe `unlock_cta_clicked` with `unlock_intent: single_episode_unlock`.
- [ ] If fake auth is included, verify `auth_started` and `auth_returned` preserve `show_id`, `episode`, `source`, and drawer return state.
- [ ] Trigger fake payment success and observe `fake_payment_started`, `fake_payment_succeeded`, `entitlement_granted`, and `same_episode_resumed` in order.
- [ ] Trigger fake payment cancel/failure and verify return to the same locked episode, not Home.
- [ ] Repeat a fake success callback with the same idempotency key and verify no duplicate fake entitlement grant is counted.
- [ ] Verify `unlocked=1` is recorded only as UX state in `same_episode_resumed`, not as authoritative entitlement.
- [ ] Inspect network requests and confirm no calls to Meta/Facebook, analytics vendors, production APIs, backend event collectors, or databases.

## 11. Network boundary verification notes

Recommended local/staging evidence package:

- Browser DevTools Network screenshot or Playwright trace showing no requests to Meta/Facebook or analytics vendors during the funnel.
- Dependency check showing no production analytics packages were added.
- Test fixture output showing event order and dedupe behavior.
- Reviewer note confirming all event sinks are local/staging/debug only.

Known repo-state signal at spec time: the repository already has a `banned-dependency` guard listing Facebook SDK/Pixel/CAPI and common analytics packages as banned dependency categories. This spec depends on retaining that boundary.

## 12. Hidden dependencies and risks

| Risk | Why it matters | Mitigation |
| --- | --- | --- |
| Event taxonomy gets mistaken for approval to add production analytics. | Production analytics is explicitly out of Phase 4C scope. | Keep all sinks local/staging and require a later approval gate before vendor integration. |
| `unlocked=1` is treated as a grant source. | URL state is forgeable and only represents post-unlock UX in the current mock. | Event `same_episode_resumed` may record `unlocked_query_state`, but `entitlement_granted` must come from fake grant state/idempotency in staging tests. |
| Free preview becomes instrumented with prompts or consent blockers. | PRD requires free preview before login/payment/PWA prompts. | Analytics capture must be passive local/staging debug only. No consent prompt may block playback in P0/Phase 4C. |
| Real ad identifiers leak into fixtures. | Attribution params are untrusted and may contain user/campaign data. | Allowlist, truncate, and use synthetic examples only. |
| Duplicate callbacks inflate conversion metrics. | Payment/auth callbacks can retry in later real systems. | Require idempotency keys and duplicate markers in staging. |
| Event order hides same-episode return bugs. | Returning to Home or losing episode context breaks the core funnel. | `same_episode_resumed.return_episode_matches` is required and should be a reviewer gate. |

## 13. Production analytics gate list

Before any production analytics, Meta Pixel, CAPI, API, or third-party analytics work begins, a new approval package must answer:

1. Which vendor(s), if any, are approved?
2. What consent UX and privacy copy are approved for target market(s)?
3. Which identifiers may be collected, retained, hashed, or transmitted?
4. What retention period and deletion process apply?
5. What server/client split is approved for events?
6. How will event dedupe map from staging keys to production IDs?
7. How will QA prove no events fire before required consent?
8. How will ad attribution be validated without unsafe query param trust?
9. What environments, secrets, dashboards, and access controls are approved?
10. What rollback/kill switch exists for production tracking?

None of these are approved by this Phase 4C spec.

## 14. Acceptance criteria for this spike

This spike is ready for reviewer gate when:

- Event names and properties cover the PRD primary funnel from ad landing through same-episode resume.
- Dedupe and idempotency behavior is defined for duplicate fake payment/grant callbacks.
- Consent/privacy notes explicitly state staging-only placeholders and no final legal posture.
- The spec forbids Meta Pixel/CAPI/API, production analytics, backend/database event ingestion, and real user/payment identifiers.
- The local/staging verification checklist can prove no production analytics call occurs.
- The spec preserves free-preview-first, single-episode unlock primary CTA, Story Pass secondary CTA, and same-episode return.

## 15. Stop conditions

Stop and request explicit user approval if any follow-up task proposes or requires:

- Real Meta Pixel, CAPI, Facebook SDK, Marketing API, Graph API, or production ad-platform integration.
- Any production analytics vendor or endpoint.
- A backend/database/event collector used by real users.
- Real login/auth, account identifiers, payment identifiers, IP/device fingerprinting, or production user data.
- Consent/legal language presented as final.
- Production deployment, secrets, DNS/cutover, or real traffic.
- Any change that sends ad traffic to Home/Search/Show Detail first, requires login before free preview, or loses the same-episode return path.

## 16. Go/no-go recommendation

Recommendation: APPROVE for Phase 4C local/staging spike planning only, with the explicit boundary that this taxonomy is a debug/test contract and not an authorization to implement production analytics.

Recommended next step: create a reviewer-gated implementation spike, if needed, that adds a local-only event recorder and tests against this contract without network egress or third-party analytics dependencies.
